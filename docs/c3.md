## 1. Overview

C3 (Context Cascade Compression) is a text compression framework that uses a cascaded encoder-decoder architecture to compress long text contexts into compact latent representations.

### 1.1 Core Innovation

The key innovation is using **learnable continuous embeddings** (not discrete tokens) as latent representations, inserted directly into the embedding space at positions marked by special tokens.

**Paper Quote (Section 3.2)**:
> "We introduce a set of learnable embeddings designated as the context query. This query is materialized as a trainable tensor Q of dimensions N×D, where N represents the fixed number of tokens desired for the output latent context, and D corresponds to the hidden dimension."

### 1.2 Design Philosophy

```
Original Text (M tokens) → Encoder → Latent Tokens (N tokens) → Decoder → Reconstructed Text
                                                    ↓
                              Compression Ratio: M/N (e.g., 40x = 1280/32)
```

**Paper Quote (Abstract)**:
> "A small LLM, acting as the first stage, performs text compression by condensing a long context into a set of latent tokens (e.g., 32 or 64 in length), achieving a high ratio of text tokens to latent tokens. A large LLM, as the second stage, then executes the decoding task on this compressed context."

**Source Location**: `third-part/C3-Context-Cascade-Compression-main/C3-master/C3/model/C3.py:21-23` (Special token definitions)

## 3. Core Components

### 3.1 Context Query (Q) - Paper's "Context Query"

```python
# third-part/C3-Context-Cascade-Compression-main/C3-master/C3/model/C3.py:35
self.Q = nn.Embedding(config.latent_token_len, config.contexts_compression_llm_hidden_size)
```

**Paper Quote (Section 3.2)**:
> "To facilitate the compression, we introduce a set of learnable embeddings designated as the context query. This query is materialized as a trainable tensor Q of dimensions N×D, where N represents the fixed number of tokens desired for the output latent context."

**Purpose**: Learnable embeddings that serve as "queries" to extract compressed representations from the encoder.

**Shape**: `[N, D]` (paper notation) where:
- N = `latent_token_len` (e.g., 32, 64)
- D = `contexts_compression_llm_hidden_size` (encoder hidden dimension, e.g., 1536 for Qwen2.5-1.5B)

**Key Insight**: These are NOT token IDs from a vocabulary. They are continuous, learnable vectors initialized randomly and optimized during training. The paper explicitly calls them "non-interpretable latent tokens."

### 3.2 MM Projector - Projection Layer

```python
# third-part/C3-Context-Cascade-Compression-main/C3-master/C3/model/C3.py:36
self.mm_projector = nn.Linear(config.contexts_compression_llm_hidden_size, config.hidden_size)
```

**Purpose**: Projects latent representations from encoder's hidden space to decoder's hidden space.

**Shape**: `[D_enc, D_dec]` where:
- D_enc = encoder hidden dimension (`contexts_compression_llm_hidden_size`)
- D_dec = decoder hidden dimension (`hidden_size`)

**Key Insight**: This enables asymmetric model pairs (smaller encoder, larger decoder). The name `mm_projector` (multimodal projector) is borrowed from VLM terminology - the code was adapted from GOT-OCR2.0.

## 4. Encoder Phase: Text → Latent Tokens

This section details the **Context Compression Encoder LLM** (paper's terminology), implemented as `llm1` in code.

### 4.1 Step 1: Tokenization and Embedding

```python

```

**Input**: `context_ids` (paper: "Input Long Context") - Token IDs of the text to compress
**Output**: `context_embeds` - Shape `[B, M, D]` (paper notation)

**Source Location**: `third-part/C3-Context-Cascade-Compression-main/C3-master/C3/model/C3.py:64`

### 4.2 Step 2: Prepare Query Embeddings (Context Query)

```python
# third-part/C3-Context-Cascade-Compression-main/C3-master/C3/model/C3.py:73-76
context_features = []
for i in range(context_embeds.shape[0]):
    context_features.append([self.Q.weight])
```

**Purpose**: Each batch item gets the same Q.weight (paper: "Context Query").
**Shape**: Each `self.Q.weight` is `[N, D]` (paper notation)

**Important**: The variable name `context_features` in code corresponds to "Context Query" in the paper. The naming discrepancy is due to code heritage from VLM architectures.

**Source Location**: `third-part/C3-Context-Cascade-Compression-main/C3-master/C3/model/C3.py:73-76`

### 4.3 Step 3: Insert Queries into Context Embeddings

```python
# C3.py:79-102
new_context_embeds = []
image_start_tokens_list = []
for cur_context_ids, cur_context_embeds, cur_context_features in zip(context_ids, context_embeds, context_features):
    if use_im_start_end:
        image_start_tokens = torch.where(cur_context_ids == im_start_token)[0]
        image_start_tokens_list.append(image_start_tokens)

        for image_start_token_pos, per_cur_image_features in zip(image_start_tokens, cur_context_features):
            per_cur_image_features = per_cur_image_features.to(device=cur_context_embeds.device)
            num_patches = per_cur_image_features.shape[0]
            if cur_context_ids[image_start_token_pos + num_patches + 1] != im_end_token:
                raise ValueError("The image end token should follow the image start token.")
            
            cur_context_embeds = torch.cat(
                (
                    cur_context_embeds[:image_start_token_pos+1],
                    per_cur_image_features,
                    cur_context_embeds[image_start_token_pos + num_patches + 1:]
                ), 
                dim=0
            )
        new_context_embeds.append(cur_context_embeds)
```

**Critical Operation**: Replace the placeholder tokens with actual query embeddings (paper: "Context Query").

**Code Variable Naming Note**:
- `image_start_tokens` / `image_start_token_pos` - These names are inherited from VLM code (GOT-OCR2.0). In C3's text compression context, they mark where Context Query embeddings are inserted.
- `per_cur_image_features` - Actually the Context Query (Q.weight), not image features.

**Before Insertion**:
```
[Text embeddings] [<img>] [<imgpad> × N] [</img>] [Remaining text]
                    ↑         ↑         ↑
                 position  replace   position
```

**After Insertion**:
```
[Text embeddings] [<img>] [Q_1, Q_2, ..., Q_N] [</img>] [Remaining text]
                          ↑─────────────────↑
                            Q.weight inserted (Context Query)
```

**Source Location**: `third-part/C3-Context-Cascade-Compression-main/C3-master/C3/model/C3.py:79-102`

### 4.4 Step 4: Encoder Forward Pass

```python
# third-part/C3-Context-Cascade-Compression-main/C3-master/C3/model/C3.py:108-114
context_embeds = torch.stack(new_context_embeds, dim=0)
llm1_hidden_states = self.llm1.forward(
    input_ids=None, attention_mask=context_attention_mask, past_key_values=None,
    inputs_embeds=context_embeds, use_cache=None, position_ids = None,
    output_attentions=output_attentions, output_hidden_states=True,
    return_dict=return_dict
)['hidden_states'][-1]
```

**Key Point**: The encoder receives `inputs_embeds` directly, NOT `input_ids`. This is crucial - the Context Query embeddings are continuous vectors, not discrete tokens.

**Output**: `llm1_hidden_states` - Shape `[B, M+N, D]` (paper notation)

**Source Location**: `third-part/C3-Context-Cascade-Compression-main/C3-master/C3/model/C3.py:108-114`

### 4.5 Step 5: Extract Latent Representations (Latent Context)

```python
# third-part/C3-Context-Cascade-Compression-main/C3-master/C3/model/C3.py:115-119
latent_contexts = []
for i, llm1_hidden_state in enumerate(llm1_hidden_states): 
    image_start_token_pos = image_start_tokens_list[i]
    llm1_hidden_state = llm1_hidden_state[image_start_token_pos+1:image_start_token_pos + num_patches+1]
    latent_contexts.append(llm1_hidden_state)
```

**Operation**: Extract the hidden states at positions where Q (Context Query) was inserted.

**Paper Quote (Figure 3 Caption)**:
> "A smaller encoder LLM compresses a variable-length Input Long Context into a fixed-length Latent Context guided by learnable Context Query tokens."

**Extraction Logic**:
- Start: `image_start_token_pos + 1` (after `<img>` marker)
- End: `image_start_token_pos + num_patches + 1` (before `</img>` marker)

**Output**: `latent_contexts` (paper: "Latent Context") - List of tensors, each `[N, D]` (paper notation)

**Source Location**: `third-part/C3-Context-Cascade-Compression-main/C3-master/C3/model/C3.py:115-119`

This section details the **Decoder LLM** (paper's terminology), which uses the Qwen2Model layers in the code.

### 5.1 Step 1: Project Latent to Decoder Space


**Purpose**: Project from encoder hidden dimension (D_enc) to decoder hidden dimension (D_dec) using `mm_projector`.

```python
if inputs_embeds is None:
    inputs_embeds = self.embed_tokens(input_ids)
```

**Input**: `input_ids` (paper: "Prompt") - Token IDs containing `<img>` placeholder
**Output**: `inputs_embeds` - Shape `[B, L, D_dec]`

### 5.3 Step 3: Insert Latent Embeddings

```python
# C3.py:129-153
new_input_embeds = []
for cur_input_ids, cur_input_embeds, cur_latent_features in zip(input_ids, inputs_embeds, latent_features):
    if use_im_start_end:
        if (cur_input_ids == im_start_token).sum() != (cur_input_ids == im_end_token).sum():
            raise ValueError("The number of image start tokens and image end tokens should be the same.")
        image_start_tokens = torch.where(cur_input_ids == im_start_token)[0]
        for image_start_token_pos, per_cur_latent_features in zip(image_start_tokens, cur_latent_features):
            per_cur_latent_features = per_cur_latent_features.to(device=cur_input_embeds.device)
            num_patches = per_cur_latent_features.shape[0]
            if cur_input_ids[image_start_token_pos + num_patches + 1] != im_end_token:
                raise ValueError("The image end token should follow the image start token.")
            cur_input_embeds = torch.cat(
                (
                    cur_input_embeds[:image_start_token_pos+1],
                    per_cur_latent_features,
                    cur_input_embeds[image_start_token_pos + num_patches + 1:]
                ), 
                dim=0
            )
        new_input_embeds.append(cur_input_embeds)
```

**Critical Operation**: Insert the projected Latent Context embeddings into the decoder input.

**Paper Quote (Figure 3 Caption)**:
> "Subsequently, a larger decoder LLM uses this compact Latent Context and a prompt to perform the downstream task, such as reconstructing the original text."

**Before Insertion**:
```
[<img>] [<imgpad> × N] [</img>] [Prompt]
   ↑        ↑          ↑
Markers for insertion positions
```

**After Insertion**:
```
[<img>] [latent_1, ..., latent_N] [</img>] [Prompt]
          ↑─────────────────────↑
           Projected Latent Context inserted
```

**Source Location**: `third-part/C3-Context-Cascade-Compression-main/C3-master/C3/model/C3.py:129-153`

### 5.4 Step 4: Stack and Forward

```python
# third-part/C3-Context-Cascade-Compression-main/C3-master/C3/model/C3.py:153-160
inputs_embeds = torch.stack(new_input_embeds, dim=0)

return super(C3QwenModel, self).forward(
```

```python

if labels is not None:
    # Shift so that tokens < n predict n
```

## 7. Inference Flow

### 7.1 Chat Interface

```python
# third-part/C3-Context-Cascade-Compression-main/C3-master/C3/model/C3.py:368-412
def chat(self, tokenizer, context, prompt):
    self.initialize_special_tokenizer(tokenizer)
    
    # Prepare decoder input with placeholder
    qs = prompt
    qs = DEFAULT_IM_START_TOKEN + DEFAULT_IMAGE_PATCH_TOKEN * self.get_model().config.latent_token_len + DEFAULT_IM_END_TOKEN + '\n' + qs
    
    # Prepare context with placeholder
    context = context + DEFAULT_IM_START_TOKEN + DEFAULT_IMAGE_PATCH_TOKEN * self.get_model().config.latent_token_len + DEFAULT_IM_END_TOKEN
    
    # Tokenize
    inputs = tokenizer([prompt])
    inputs_context = tokenizer([context])
    input_ids = torch.as_tensor(inputs.input_ids).cuda()
    inputs_context_ids = torch.as_tensor(inputs_context.input_ids).cuda()
    
    # Generate
    output_ids = self.generate(
        input_ids,
        context_ids=inputs_context_ids,
        do_sample=False,
        num_beams=1,
        max_new_tokens=4096,
        ...
    )
```

**Inference Process**:
1. Append `<img><imgpad>×N</img>` to context text
2. Prepend `<img><imgpad>×N</img>` to prompt
3. Tokenize both separately
4. Pass to model (encoder processes context, decoder generates from prompt + latent)

**Source Location**: `third-part/C3-Context-Cascade-Compression-main/C3-master/C3/model/C3.py:368-412`

## 8. Key Design Decisions

### 8.1 Why Embeddings, Not Token IDs?

| Approach            | Pros                                 | Cons                                      |
|---------------------|--------------------------------------|-------------------------------------------|
| **Token IDs**       | Simple, interpretable                | Limited vocabulary, discrete optimization |
| **Embeddings (C3)** | Continuous, flexible, differentiable | Less interpretable                        |

**Decision**: C3 uses **continuous embeddings** because:
1. Latent representations are not discrete vocabulary items
2. Enables end-to-end gradient flow
3. No vocabulary bottleneck (can represent any continuous vector)

**Paper Quote (Section 2, Memory and Summarization)**:
> "Rather than generating a human-readable text summary, we train a dedicated encoder LLM to directly distill textual information into a fixed-length set of non-interpretable latent tokens."

### 8.2 Why Separate Encoder and Decoder (Cascaded Architecture)?

```python
# third-part/C3-Context-Cascade-Compression-main/C3-master/C3/model/C3.py:37
self.llm1 = None  # Encoder LLM (paper: "Context Compression Encoder LLM") - attached later
```

**Source Location**: `third-part/C3-Context-Cascade-Compression-main/C3-master/C3/model/C3.py:37`

**Benefits**:
- Asymmetric model sizes (small encoder, large decoder)
- Different hidden dimensions (via mm_projector)
- Memory efficiency (encoder can be frozen after pretraining)

**Paper Quote (Section 3.1)**:
> "C3 employs a cascaded two-LLM architecture, comprising a context compression encoder LLM and a decoder LLM. Specifically, we use Qwen2.5 1.5B as the context compression encoder and Qwen2.5 3B as the decoder."

### 8.3 Why Special Tokens for Insertion?

**Alternative**: Direct concatenation at fixed positions.

**C3's Choice**: Special tokens (`<img>`, `</img>`) provide:
1. Flexibility in tokenization (can appear anywhere in the sequence)
2. Clear boundaries for extraction
3. Compatibility with existing tokenizer infrastructure

**Note**: The token names `<img>`, `</img>`, `<imgpad>` are code-specific and inherited from GOT-OCR2.0/VLM codebases. The paper does not specify particular token names.

### 8.4 Why Context Query (Q) is Shared Across Batch?

```python
# third-part/C3-Context-Cascade-Compression-main/C3-master/C3/model/C3.py:74-76
for i in range(context_embeds.shape[0]):
    context_features.append([self.Q.weight])  # Same Q.weight for all batch items
```

**Reason**: 
- The Context Query (Q.weight) contains learnable position encodings
- Same query embeddings for all samples in a batch
- The encoder learns to respond to these fixed query positions
- Analogy: Like learnable position embeddings in transformers - shared across all sequences

### 8.5 Why mm_projector (Projection Layer)?

```python
# third-part/C3-Context-Cascade-Compression-main/C3-master/C3/model/C3.py:36
self.mm_projector = nn.Linear(config.contexts_compression_llm_hidden_size, config.hidden_size)
```

**Purpose**: Enable **model cascade** - use different model sizes for encoder and decoder.

**Paper Example (Section 3.1)**:
- Encoder: Qwen2.5-1.5B (D=1536) → smaller, faster
- Decoder: Qwen2.5-3B (D=2048) → larger, higher quality generation

**Note**: The name `mm_projector` (multimodal projector) comes from VLM terminology. In C3, it projects between encoder and decoder hidden spaces.

## 9. Summary: The Complete Pipeline

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    ENCODER PHASE (Context Compression Encoder LLM)          │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  context_ids [B, M]  ←── Paper: "Input Long Context"                        │
│       │                                                                     │
│       ▼ embed_tokens() ─────────────── C3/model/C3.py:64                   │
```

## 10. Code Heritage Note

The C3 implementation is adapted from GOT-OCR2.0, which is a Vision-Language Model. This explains:

1. **Special Token Names**: `<img>`, `</img>`, `<imgpad>` are image-related tokens in VLMs, repurposed for latent token insertion in C3.

2. **Variable Names**: 
   - `image_start_tokens`, `image_start_token_pos` → actually mark latent insertion positions
   - `per_cur_image_features` → actually Context Query (Q.weight)
   - `mm_projector` (multimodal projector) → actually encoder-to-decoder projection

3. **Code Comments**: Some comments reference "image" terminology from the original VLM codebase.
